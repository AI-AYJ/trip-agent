# 요구사항 문서

## 소개

선택한 지역 기반 장소 추천 에이전트 기능을 구현합니다.
본 시스템은 자동으로 교통 경로를 계산하거나 하루 일정을 최적화하는 방식이 아니라,

👉 **지역 선택 → 장소 추천 → 사용자 선택 → 지도에 즐겨찾기 표시**

흐름으로 동작합니다. 사용자는 추천된 장소 후보 중 원하는 장소를 직접 선택하여 자신만의 방문 후보 리스트를 만들 수 있습니다. 추천되는 장소는 영업 시간 및 현재 영업 여부를 고려합니다.

---

## 용어 정의

- **State**: 사용자의 현재 상태 (시간, 피로도, 동행 여부, 관심사, 선택한 지역)
- **Place**: 추천되는 장소 (관광지, 카페, 맛집 등)
- **Favorite**: 사용자가 선택한 장소 (즐겨찾기)
- **Map Marker**: 지도에 표시되는 장소 위치
- **Agent**: 특정 역할을 수행하는 기능 단위
- **Operating Status**: 특정 시간 기준 장소의 영업 여부
- **Storage**: 사용자의 선택 장소 또는 상태를 저장하는 저장소

---

## 요구사항

### 요구사항 1: 현재 상태 및 지역 입력

**사용자 스토리:** 사용자로서, 내가 선택한 지역과 현재 상태에 맞는 장소를 추천받고 싶습니다.

#### 인수 기준

1. THE App은 시작 시간, 피로도, 동행 여부, 관심사, 선택한 지역을 입력받아야 한다.
2. THE State Agent는 입력된 시간, 피로도, 동행 여부, 관심사, 선택한 지역을 해석해야 한다.
3. THE State Agent는 활동 가능 범위와 추천 기준을 계산해야 한다.
4. WHEN 사용자가 지역을 선택하지 않은 경우, THE App은 장소 추천을 수행하지 않아야 한다.
5. WHEN 피로도가 높은 경우, THE App은 가까운 장소 또는 부담이 적은 장소를 우선 추천해야 한다.

---

### 요구사항 2: 장소 추천

**사용자 스토리:** 사용자로서, 선택한 지역 안에서 지금 방문 가능한 장소를 추천받고 싶습니다.

#### 인수 기준

1. THE Place Agent는 사용자가 선택한 지역을 기반으로 장소를 추천해야 한다.
2. THE Place Agent는 선택한 지역 내에서만 장소 후보를 생성해야 한다.
3. THE Place Agent는 관광지, 카페, 맛집을 포함한 장소 후보를 생성해야 한다.
4. THE Place Agent는 최소 10개의 장소를 추천해야 한다.
5. THE Place Agent는 사용자의 관심사와 일치하는 장소를 우선 추천해야 한다.
6. THE Place Agent는 각 장소의 영업 시간을 확인해야 한다.
7. WHEN 현재 시간 기준 영업 중이 아닌 경우, THE App은 해당 장소를 추천 후보에서 제외하거나 영업 종료 상태로 명확히 표시해야 한다.
8. WHEN 영업 종료 시간이 임박한 경우, THE App은 해당 장소의 우선순위를 낮춰야 한다.
9. THE Place Agent는 각 장소에 대해 이름, 카테고리, 위치, 간단 설명, 영업 여부를 제공해야 한다.

---

### 요구사항 3: 사용자 선택 및 즐겨찾기

**사용자 스토리:** 사용자로서, 추천된 장소 중 마음에 드는 장소만 선택하고 싶습니다.

#### 인수 기준

1. THE App은 추천된 장소 리스트를 사용자에게 표시해야 한다.
2. THE User는 추천된 장소 중 여러 개를 선택할 수 있어야 한다.
3. WHEN 사용자가 장소를 선택하는 경우, THE App은 해당 장소를 Favorite에 추가해야 한다.
4. WHEN 사용자가 선택을 취소하는 경우, THE App은 해당 장소를 Favorite에서 제거해야 한다.
5. THE Favorite 리스트는 사용자가 선택한 장소만 포함해야 한다.
6. THE App은 이미 선택된 장소를 시각적으로 구분해야 한다.

---

### 요구사항 4: 지도 표시

**사용자 스토리:** 사용자로서, 내가 선택한 장소들을 지도에서 한눈에 보고 싶습니다.

#### 인수 기준

1. THE App은 Favorite 장소를 지도에 표시해야 한다.
2. THE Map은 선택된 장소를 Marker 형태로 표시해야 한다.
3. THE Marker는 장소 이름, 카테고리, 영업 여부를 포함해야 한다.
4. WHEN 사용자가 장소를 선택하는 경우, THE Map은 해당 장소의 Marker를 추가해야 한다.
5. WHEN 사용자가 선택을 취소하는 경우, THE Map은 해당 장소의 Marker를 제거해야 한다.
6. THE Map은 사용자가 선택한 지역을 기준으로 초기 화면을 표시해야 한다.
7. THE Map은 카카오 맵 API(`appkey=a072a7509eb4792590bbd3a1edb09d00`)를 사용해야 한다.

---

### 요구사항 5: 선택 결과 출력

**사용자 스토리:** 사용자로서, 내가 선택한 장소들을 정리된 형태로 확인하고 싶습니다.

#### 인수 기준

1. THE App은 Favorite 리스트를 출력해야 한다.
2. THE Favorite 리스트는 사용자가 선택한 순서를 유지해야 한다.
3. THE Favorite 리스트는 장소 이름, 카테고리, 영업 여부를 포함해야 한다.
4. THE App은 단순 추천 결과가 아니라 사용자 선택 기반 결과를 제공해야 한다.

---

### 요구사항 6: 상태 저장

**사용자 스토리:** 사용자로서, 선택한 장소가 새로고침 후에도 유지되기를 원합니다.

#### 인수 기준

1. WHEN 사용자가 장소를 Favorite에 추가하는 경우, THE Storage는 선택 상태를 저장해야 한다.
2. WHEN 사용자가 Favorite에서 장소를 제거하는 경우, THE Storage는 변경된 선택 상태를 저장해야 한다.
3. WHEN App이 초기화되는 경우, THE App은 Storage에서 Favorite 리스트를 복원해야 한다.
4. THE Storage는 선택한 지역 정보도 함께 저장할 수 있어야 한다.

## 병목의 원인

1. **Gemini API 응답 지연**: 지역 내 장소 추천 1회 요청 시 평균 3~8초 소요. 네트워크 상태에 따라 최대 15초까지 증가. 사용자가 결과를 기다리는 동안 이탈 가능성 높음.
2. **영업 시간 판단 복잡도**: 장소별 open/close 시간을 현재 시각과 비교하는 로직이 자정 넘김(예: 22:00~02:00) 케이스를 처리해야 하며, 잘못 판단 시 영업 종료 장소를 추천하는 실패 비용 발생.
3. **즐겨찾기 ↔ 마커 동기화 비용**: 즐겨찾기 1회 토글 시 카카오 맵 마커 추가/제거 + FavoritePanel 리렌더링이 동시 발생. 즐겨찾기 10개 이상일 때 마커 전체 재생성 방식은 100ms 이상 지연 유발.
4. **AI 판단이 필요한 지점 (룰베이스로 불가)**: 피로도·동행 여부·관심사 조합에서 "어떤 장소가 더 적합한가"는 단순 if-else로 결정 불가. 예: 피로도 HIGH + 동행 FAMILY + 관심사 액티비티 → 실내 체험형 장소 우선이지만, 관심사가 공원이면 야외 저강도 코스 우선. 이 판단을 Gemini 프롬프트에 컨텍스트로 전달해야 함.
5. **추천 장소 10개 미만 케이스**: 소도시·야간 시간대에서 영업 중인 장소가 10개 미만일 수 있음. 이 경우 사용자에게 이유를 안내하지 않으면 "버그"로 인식됨.

## 암묵 규칙 및 예외 처리

1. 사용자가 선택한 지역 밖의 장소는 추천하지 않는다. Gemini 응답에 지역 외 장소가 포함된 경우 클라이언트에서 필터링하지 않고 프롬프트 재설계로 방지한다.
2. 현재 영업 중인 장소를 우선 추천한다. 영업 여부는 `operatingHours.open`~`operatingHours.close` 구간으로 판단하며, 자정 넘김(close < open)은 +24시간 보정으로 처리한다.
3. 영업 종료 시간까지 1시간 미만 남은 장소는 "마감 임박"으로 표시하고 우선순위를 낮춘다.
4. 피로도 HIGH일 때 Gemini 프롬프트에 "실내, 앉아서 즐길 수 있는, 이동 거리 짧은" 조건을 명시적으로 추가한다.
5. 동행 FAMILY·FRIENDS일 때 Gemini 프롬프트에 "개인 취향보다 공통 만족도 높은" 조건을 추가한다.
6. 추천 장소가 10개 미만일 경우, 가능한 장소만 표시하고 UI에 "현재 시간 기준 영업 중인 장소가 N개입니다" 메시지를 표시한다.
7. 사용자가 Favorite에 추가한 장소는 새 추천 결과가 와도 자동으로 제거하지 않는다.
8. Gemini API가 유효하지 않은 JSON을 반환하면 빈 배열로 처리하고 "추천 결과를 불러오지 못했습니다. 다시 시도해주세요." 메시지를 표시한다.

## 실패 비용

1. **지역 외 장소 추천**: 사용자가 "강남구"를 선택했는데 "홍대" 장소가 나오면 신뢰도 즉시 하락. 재추천 요청 발생 → Gemini API 추가 호출 비용.
2. **영업 종료 장소 추천**: 사용자가 현장에 도착했을 때 문이 닫혀 있으면 앱 삭제 수준의 불만. 영업 시간 판단 오류는 허용 불가.
3. **추천 결과 0개 또는 안내 없는 10개 미만**: 사용자가 "버그"로 인식. 이유 안내 없이 빈 화면이 나오면 이탈률 급증.
4. **즐겨찾기 장소가 지도에 미표시**: 핵심 기능 실패. 마커 수 != favorites 수이면 사용자가 선택한 장소를 지도에서 찾을 수 없음.
5. **새로고침 후 즐겨찾기 소실**: 사용자가 선택한 장소 목록이 사라지면 재선택 필요 → 앱 신뢰도 하락.
6. **마커 좌표 오류**: Gemini가 반환한 lat/lng가 실제 장소와 다를 경우 지도에서 엉뚱한 위치에 마커 표시 → 방문 계획 혼란.

## 핵심 제약

1. 교통 경로 자동 계산 기능은 포함하지 않는다.
2. 자동 일정 최적화 기능은 포함하지 않는다.
3. 추천 결과는 선택한 지역을 기준으로 제한한다.
4. 추천 장소는 영업 시간 및 영업 상태를 고려해야 한다.
5. 최종 결과는 사용자가 직접 선택한 Favorite 장소를 지도에 표시하는 방식으로 제공한다.
6. 지도는 카카오 맵 API를 사용한다 (Leaflet 사용 금지).

---

## 한 줄 정의

본 시스템은 사용자의 현재 상태와 선택한 지역을 기반으로 장소를 추천하고, 사용자가 선택한 장소를 즐겨찾기 형태로 관리하며, 선택된 장소를 카카오 지도에 표시하는 인터랙티브 장소 추천 에이전트이다.
